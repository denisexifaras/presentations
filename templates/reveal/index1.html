{{notes}} -When we talk about big data typically we are referring data that is either high-dimensional and/or >>GB in size when zipped(!) -need both space and hardware that will support input/output - multiple users accessing data, transferring it around and writing out data, simultaneously. (this is a real problem - bring up WTCHG) Where do you store it? -who will need to access it? obviously we can't know this definitely but it's a legitimate concern, especially as everything becomes more and more interdisciplinary. -people access to the same server? or should it be online? Is it even downloadable? Need an ftp service? How long would it take to transfer files? -security? Could you solve your storage problem on the cloud? ... native american example -Looking ahead, this is a solvable problem.

  2. How?

-how do you store it? All in one blob? -relational or non-relational: many files, many directories? -how many people will need to access at any one time? -will they need to access all of it or should it very be straightforward to pull out sub-chunks? - leq 2K files/directory -format? Json - great for web design but would be a really bad idea for genomic data for instance: in my current project I use genomic data for 5000 individuals at 1M DNA positions. Not only do I know what their DNA is, but how confidently it was obtained in the lab (so for each person I have 1M of these scores too), I have a bunch of measurements for specific genes that I am interested, their gender, whether they smoke, their age and dietery habits. Storing this in json format - I have not calculated but would argue with conviction - is impossible.) -again, who will need to use it? What software do they use? What analysis will they perform? -csv vs vcf vs plain text vs json -should always be such that it allows for parallelisation of jobs on cluster -of course it's not just accessing and managing but writing out afterwards as well... -this is a solvable problem.
